Section 1 — Introduction (0:00–0:45)

HAL Guardian is a proof-of-concept, local-first security assistant. It is running Google's Gemma 4 model through Ollama on this machine, analyzing code and untrusted input without sending your data to a cloud API.

In this demo you will see the main modules in the order they appear in the sidebar: Code Guardian, Trust Shield, Model Playground, Subagent Console, Audit Engine, and Health. You will also see the Settings and Manual pages, which are administrative and reference pages rather than active tools.

Everything you see is running locally. The model endpoint is 127.0.0.1:11434. There are no API keys, no telemetry, and no remote inference by default. Audit entries, saved prompts, webfetch settings, and the whitelist are persisted locally with SQLite ("ess-kyoo-light") and JSONL ("jay-son-el") files, so they survive application restarts.

Section 2 — Architecture and Security Model (0:45–1:40)

The architecture is straightforward. The front end is a Streamlit application. The back end is a set of Python modules that build prompts, call Ollama, parse responses, log actions, and persist settings.

The main pages map to those modules in order. Code Guardian lets you review source code. Trust Shield lets you scan prompts and files. Model Playground is a free-form chat and model comparator that lets you compare how different models answer the same prompt, which is useful for choosing the best model for a specific job. The Subagent Console exposes every module as a callable command. Audit Engine shows the local logs, and Health shows environment status. Settings controls the optional webfetch feature, and Manual provides reference documentation.

Persistence is handled through JSONL ("jay-son-el") files and SQLite ("ess-kyoo-light"). Audit entries, saved prompts, webfetch settings, whitelist entries, model comparison history, and the complete audit trail are stored locally and survive application restarts.

The security model is built on a few simple rules: 1) Inference stays local. 2) Audit logs stay local. 3) Model output is never executed automatically. 4) Webfetch is disabled by default and gated by a whitelist and an explicit confirmation step, so fetched content is never passed to the model without your approval.

Section 3 — Code Guardian (1:40–2:45)

Code Guardian lets you review source code, pasted text, or fetched web content. The module reads the file, selects a system prompt based on the detected language, and sends a structured request to Gemma 4.

The model returns a Markdown review. A lightweight parser extracts a verdict, a summary table, and individual findings. The full Markdown response is always preserved for verification.

In this example, we are reviewing bad_login.php, a sample file that ships with the application and contains intentional security issues.

[Jump cut to results]

The verdict is "needs changes." The summary table breaks findings into security, testing, complexity, and style categories. Each finding includes a severity, line reference, description, and recommendation.

For medium-and-above findings, you can request a suggested fix. This sends only the relevant code block back to the model with a repair prompt. As with all generated content, the fix is displayed for your review and is not applied automatically.

The full review can be exported as JSON or Markdown, so another script or agent can consume the structured result or the original Markdown. This is useful if you want to feed findings into a CI pipeline, a ticket system, or a secondary review process.

Section 4 — Trust Shield (2:45–3:45)

Trust Shield lets you scan prompts, emails, documents, and web pages for signs of manipulation or hidden commands. It combines deterministic pattern matching with an optional deep scan from Gemma 4.

The quick scan detects command language, meta-instructions, and encoded payloads such as Base64, hex, and URL-encoded strings. It decodes the payload automatically.

In this example, the input is the suspicious_email.txt file that ships with the application. It contains a Base64 string. The quick scan decodes it and flags the underlying destructive command. The trust level is marked suspicious.

The deep scan sends the same input to Gemma 4 for a second opinion on intent, severity, confidence, and recommended action. Because this is a local call, the analysis adds latency but not exposure.

At the bottom of the report, the sanitized text panel shows a redacted version of the input that you can share without leaking sensitive or dangerous content.

Each report can also be exported as JSON or Markdown, so the result can be consumed by another script or agent.

Section 5 — Subagent Console (3:45–4:15)

HAL Guardian is designed to be called by another AI agent, not just used by a human. The Subagent Console exposes every module as a command that returns a standard JSON envelope. Each response has the same fields: ok, agent, command, target, timestamp, duration_ms, data, and error.

This means your main AI agent can hand off work to HAL Guardian. For example, before accepting generated code, the agent can call review_code. Before fetching a URL, it can call scan. Before trusting a health report, it can call health.

The project includes an agent prompt guide that tells an external AI how to invoke HAL Guardian with short, predictable commands. You do not need to write a long prompt yourself.

Here, you see review_dir data/sample_code in action. This batch-reviews every file in the directory and returns a JSON array of results.

Section 6 — Model Playground (4:15–4:55)

The Model Playground is a free-form chat interface for any local Ollama model — like the conversation you are having with me now, but with the model running entirely on your machine. The active model is selected from the sidebar and is shared across Code Guardian, Trust Shield, and the playground.

The gem feature here is the model comparator. After a response arrives, the comparator can run the same prompt through a second local model and show both responses side by side. It displays length, latency, similarity, and an AI judge summary. This is useful for choosing the best model for a specific job, because you can see exactly how different models handle the same prompt under the same temperature.

Here you load the river-crossing reasoning puzzle at a higher temperature to stress-test model behavior. [Jump cut to response] The active model returns a correct step-by-step solution.

You then choose a second model and click Compare. [Jump cut to comparison] The comparator opens both responses side by side, shows the similarity score, and highlights basic metrics. In our test, the active Gemma 4 model solved the puzzle correctly while the comparison model misunderstood the constraints and produced irrelevant speculation. Finally, the AI judge summarizes the key differences between the two outputs.

Section 7 — Webfetch in Action (4:55–5:25)

The optional webfetch feature lets Code Guardian and Trust Shield analyze live URLs. Because of the risks involved, it requires explicit enabling, a domain whitelist, and your confirmation before any fetched content reaches the model. It only works if the machine running HAL Guardian has internet access.

In this example, you fetch a public test page hosted on our website. The domain is whitelisted in Settings, and you confirm the fetch before scanning.

The HTML is stripped to text locally, and Trust Shield scans the extracted content the same way it would scan pasted text. The same workflow applies to Code Guardian, allowing review of live documentation or sample pages.

Section 8 — Audit Engine and Conclusion (5:25–5:50)

Every action shown in this demo has been logged locally. The Audit Engine displays the JSONL ("jay-son-el") log and mirrors it to SQLite ("ess-kyoo-light") for structured querying.

Each entry records the action type, target, model, duration, success flag, and metadata. The full log can be exported as JSON or CSV for external analysis.

HAL Guardian is a proof-of-concept, not a production security product. It demonstrates that local open models can provide code review, input classification, and audit logging while keeping data on the device. The source code and demo pages are available at the links shown.
